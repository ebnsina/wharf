Install

wharf is a single binary. It needs Go 1.24 or newer to build.

go install github.com/ebnsina/wharf/cmd/wharf@latest

State lives in ~/.wharf — the manifests, the log files, and a backup of every config wharf has edited. Set WHARF_HOME to move it.

Getting started

Point wharf at a directory of projects. It reads each one and writes a manifest.

$ wharf scan ~/code
$ wharf doctor     # what will break, before you start anything
$ wharf tui        # the dashboard

Detection reads what is already there:

  • Go — go.mod, .air.toml, Makefile targets, config/config.yaml
  • Node — package.json scripts, the lockfile (so it uses the right package manager), the framework’s port
  • Python — pyproject.toml, alembic.ini, the ASGI entrypoint

It also separates the things you run from the things you only build. An SDK with no dev server is a library: it never gets a port and never appears as something to start.

Detection is a starting point, not an authority. Edit any manifest in ~/.wharf/services/ to correct it. A rescan keeps your edits.

Berths

A berth is a port a service owns. The name is the point: a berth is a numbered slot, and two ships cannot occupy one.

Detected ports collide — several services ship with :8080, several frontends with :5173. wharf assigns each service a port nobody else has, and writes it into that service’s own config. 

$ wharf berth --dry-run
✓ billing-api   app.port: 8080 → 8105    config/config.yaml
✓ dashboard     addr: ":8090" → ":8101"   configs/config.yaml

dry run — 2 files would change, none were written

What it writes, and what it will not

  • Only the port value changes. Comments, ordering and formatting survive — the edit is made at the exact position of the value, not by re-serialising the file.
  • The shape is preserved. A field holding ":8080" gets ":8105", not 8105 — the second is not a listen address and the service would not start.
  • It is idempotent. Running it twice writes nothing the second time.
  • The original is backed up to ~/.wharf/backups/ before the first write.
A config committed to git is never touched. A berth is a choice about your machine; writing it into a shared file would land it in everyone else’s checkout. wharf skips it, says so, and leaves that service on its declared port. Pass --force if you mean it.

Because the berth lives in the service’s own config, the fix is durable: run air by hand, outside wharf, and the service still comes up on the right port.

Databases

Your config already says what it wants — a Postgres at localhost:5432 holding a database called billing. wharf makes that true, rather than inventing its own connection details and rewriting your config. The DSN is the specification; provisioning conforms to it. 

$ wharf infra
● postgres    localhost:5432    up
    ✓ billing              billing-api
    ✗ auth                 missing — auth-api needs it
✗ clickhouse  localhost:9000    down
    needs events

wharf infra up starts what is missing, asking whether to run it in Docker or on your machine, waits until it actually accepts connections, and then creates the databases.

Servers, not services

Datastores are grouped by address, because that is how they exist: one Postgres holds a database for each of six services. Provisioning one container per service would start six Postgres servers — and none of them on the port your configs name.

A port held is not a port free. If something is listening but wharf cannot authenticate, it says so rather than trying to start a container on a port already in use — which fails with a Docker networking error that tells you nothing about the actual problem, which was a password.

Links

A frontend reaches its API through a hard-coded URL in a .env or a Vite config. When the API moves off a contested port, that URL still names the old one — so the app calls whatever now answers there, or nothing at all. Nothing fails loudly. The dev server starts, the page loads, and the requests go to the wrong place. 

$ wharf link
✓ console    localhost:8080 → localhost:8105   .env:4  (billing-api)
✓ dashboard  localhost:8090 → localhost:8101   web/.env:2  (itself)

Only the host:port token is rewritten. The scheme, the path, the quoting and the comments are left alone.

Working out which service a URL meant

  • One owner — if only one service ever declared that port, it is unambiguous.
  • Itself — a Vite config in web/ proxying to the Go server in the same repo means that server.
  • A sibling — acme-web reaching an API means acme-api, not an unrelated service that happened to ship with the same port.
  • The variable’s name — VITE_BILLING_API_URL means the billing service, whatever port it now uses.
Where none of that is enough, wharf stops. A bare VITE_API_URL naming a port four services shipped with is genuinely ambiguous, and a wrong guess sends a frontend’s traffic to the wrong backend — where it looks like a bug in the backend. Say it once and wharf keeps it right forever: 
$ wharf link set console VITE_API_URL=billing-api

Gateway

If a local reverse proxy fronts your services, wharf generates its config from the same manifests that assign the berths. A route can no longer name a port a service is not on, because nobody types the port twice. 

$ wharf gateway import /opt/homebrew/etc/nginx/servers/default.conf
✓ /v1/billing/   → billing-api :8105   (was :8080, now on its berth)

$ wharf gateway apply

Import matches each route to the service that originally declared its port, not to whoever holds that port now — otherwise a route would be handed to whichever service won the collision.

Both nginx and caddy drivers exist. Caddy is recommended: snippets remove the per-route boilerplate, WebSockets need no directives, and local HTTPS is automatic.

Running

$ wharf up billing-api       # the service, its deps and its datastores
$ wharf up --infra            # provision anything missing first
$ wharf logs -f               # follow its output
$ wharf db                    # psql, connection string already filled in

Nothing is left holding a port

A dev command is never one process: air compiles and then runs your binary; pnpm dev reaches Vite through a shell. Signalling only the process wharf started leaves the grandchild alive, still holding the port — and the next start fails with address already in use, which looks unrelated to the cause. Every service runs as a process group and is stopped as one.

Health is measured, not assumed

A TCP dial only proves a port is open, and a Go server binds its listener before its dependencies are wired. wharf reads the health endpoint out of your code — one service exposes /healthz, another /v1/health-check — and probes that. Where a service exposes none, it says port only rather than implying it checked something.

The dashboard

↑ ↓Move between services. Click works too.
s / x / rStart, stop, restart.
/Filter by name.
tabMove focus between the list and the log.
?Keys, and what each status means.

A berth held by a process wharf did not start gets its own status. Calling it stopped would send you hunting for a bug in the service, rather than for whatever is already sitting on its port.

Manifests

One YAML file per service, under ~/.wharf/services/. Generated by scan, then yours to correct. 

name: billing-api
path: /Users/you/code/billing-api
kind: service
stack: go
berth: 8105
declared_berth: 8080      # what it used to ask for

processes:
  - name: api
    cmd: air
    primary: true
  - name: worker
    cmd: air -c .air.worker.toml
    autostart: false      # start it from the dashboard when you need it

config:
  - format: yaml
    path: config/config.yaml
    port_key: app.port
    port_template: "{port}"

needs:
  - type: postgres
    dsn: postgres://postgres@localhost:5432/billing

health:
  type: http
  path: /v1/health-check

links:
  VITE_API_URL: console   # which service a variable points at

lifecycle:
  migrate: go run ./cmd migrate up

declared_berth is the port the project originally asked for. It is kept because the outside world still refers to services by it — an old gateway route, a URL in a .env — and matching those against the current occupant of that port would hand them to whichever service won the collision.

Commands

Anything that takes a service name defaults to the project you are standing in.

wharfInside a project: everything about it.
wharf scan <dir>Discover projects and write manifests.
wharf lsServices, berths, what is running.
wharf doctorContested ports, missing configs, dead upstreams.
wharf tuiThe dashboard.
wharf upStart a service, its dependencies and its datastores.
wharf logsWhat it printed. -f to follow.
wharf dbA database shell, already connected.
wharf berthWrite each service’s berth into its config.
wharf linkRepoint the URLs one project uses to reach another.
wharf infraDatastores, and the databases inside them.
wharf gatewayGenerate the reverse-proxy config.
wharf bootstrapFresh clone → config, install, migrate, seed.
