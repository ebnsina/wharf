Local development, for a workspace that outgrew you

Your services all want :8080. Only one can have it.

wharf reads how every project in your workspace runs, gives each one a port it owns — a berth — and starts it with its config, its databases and everything it depends on.

$go install github.com/ebnsina/wharf/cmd/wharf@latest
Berth board contested → allocated
:8080 kormi.ai-api docked
:8080 :8105 vidinfra-live-api contested docked
:8080 :8107 vidinfra-live-worker contested docked
:8080 :8108 vidinfra-livestream-api contested docked

Four services shipped with the same port. wharf writes each one’s berth into its own config, and the gateway and every frontend URL follow it there.

The knowledge lives in your head, and it is wrong

A workspace of microservices accumulates facts nobody wrote down — and some of them contradict each other.

Contested

Ports collide

Four services ship with :8080. They cannot run together, and nothing says so until one fails to bind.

Forgotten

Every service starts differently

air here, pnpm dev there, a Makefile target you have to open the Makefile to remember.

Silent

URLs drift

A frontend still calls the port its API left. Nothing fails loudly — the requests just go somewhere else.

It reads your code, not a config you have to write

Point it at a directory. It works out what is there — the run command, the port, the databases, the migration — and writes a manifest you can correct. 

$ wharf scan ~/code

  auth-api        go      service  :8103   postgres, redis
  billing-api     go      service  :8105   postgres
  console         node    service  :8100
  search          python  service  :8000   clickhouse
  player-sdk      node    library  —

! 3 services were moved off a contested port
    billing-api      8080 → 8105  (8080 already claimed by auth-api)

✓ wrote 5 manifests

Then it makes the manifest true

Each step depends on the one before it. wharf cannot route to a service before it has a berth, and it cannot migrate a database that does not exist.

1

Give each service a port it owns

The berth is written into the service’s own config — the local, gitignored one — so the fix holds even when you run the service by hand. Only the port value changes; comments and formatting survive. 

$ wharf berth
✓ billing-api   app.port: 8080 → 8105   config/config.yaml
! live-worker   skipped — config.yaml is committed to git
2

Make the databases real

Your config already says it wants a Postgres at :5432 holding a database called billing. wharf makes that true — in Docker or on your machine, whichever you pick — instead of inventing its own connection details. 

$ wharf infra up
! clickhouse is not running at localhost:9000
    search needs the database events

  [d] docker   clickhouse/clickhouse-server — wharf runs and keeps it
  [l] local    brew services start clickhouse

  choice [d]: d
✓ clickhouse is up at localhost:9000
✓ created database events for search
3

Point everything at the new address

A frontend reaches its API through a hard-coded URL in a .env. When the API moves, that URL still names the old port — so wharf repoints it, and regenerates the gateway from the same manifests. A route can no longer name a port a service is not on, because nobody types the port twice. 

$ wharf link
✓ console   localhost:8080 → localhost:8105   .env:4  (billing-api)
! web       .env:2 names :3000, declared by three services
    say which once: wharf link set web VITE_API_URL=console
4

Start it, and watch it

Services run as process groups, so nothing is orphaned holding a port when you quit. The dashboard shows what is up, what is still compiling, and what is sitting on a berth that wharf did not start. 

$ wharf up billing-api
✓ postgres ready       localhost:5432/billing
✓ billing-api          http://localhost:8105

$ wharf tui          the dashboard: start, stop, live logs

Inside a project, it already knows which one you mean

Standing in a directory is a statement about which service you are talking about. wharf does not make you say it again. 

$ cd ~/code/billing-api && wharf

  billing-api  running
  ~/code/billing-api

  stack      go
  url        http://localhost:8105   (moved off :8080 — it was contested)
  run        air
  health     GET /v1/health-check
  migrate    go run ./cmd migrate up

  ● postgres   localhost:5432   up
  ● redis      localhost:6379   up

Try it on the mess you already have

wharf knows nothing about any particular workspace. Point it at a directory and it detects what is there — a new machine, a client’s repo, the same.

$go install github.com/ebnsina/wharf/cmd/wharf@latest
